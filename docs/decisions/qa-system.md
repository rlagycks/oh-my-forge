# QA 시스템 설계 결정

## 목표

한 번의 호출로 실제 E2E 테스트를 돌리고, 발생하는 에러·버그·엣지케이스를 전부 수집한 뒤,
근본 원인을 분석해 리포트로 출력한다. 단순히 테스트를 통과시키는 게 아니라 근본 원인을 고치는 것이 목표.

구현: `skills/e2e-rca/SKILL.md`, `commands/qa-loop.md`

---

## 핵심 결정

### 분석과 수정은 분리한다

**결정:** `/qa-loop`은 Phase 5(최종 리포트)에서 멈춘다. 코드를 자동으로 수정하지 않는다.

**이유:** 개발자가 리포트를 검토하고 어떤 수정을 적용할지 결정해야 한다.
자동 수정은 근본 원인을 잘못 판단했을 때 더 큰 문제를 만든다.

**워크플로우 경계:**
```
[e2e-rca 범위]                    [개발자 결정]
자동감지 → 실행 → 분석 ──────  리포트 검토 → 수정 → 검증
                       STOP ↑
```

---

### 페르소나 기반 시나리오 생성

**결정:** 모든 경우를 테스트하지 않는다. Role × State 매트릭스를 만들고 의미있는 교차점만 선택한다.

**페르소나 = Role + State**
```
예시:
Guest         — 비인증
New User      — 인증, 데이터 없음 (첫 로그인)
Active User   — 인증, 데이터 있음
Expired User  — 인증, 구독/세션 만료
Admin         — 상승된 권한
```

**시나리오 선택 기준:**
- `✗` 셀 (접근 제어 경계) — 버그가 가장 많이 숨어있는 곳
- `empty✓` 셀 (빈 상태) — 두 번째로 흔한 버그
- HIGH 우선순위 플로우의 `✓` 셀만

**스킵:**
- LOW 우선순위 플로우
- 이미 유닛 테스트로 커버된 로직

**페르소나 자동 발견:** Phase 0.5에서 코드베이스를 grep해서 역할을 추론한다.
수동 설정 불필요. 결과는 `docs/qa/personas.md`에 저장.

---

### Playwright 설정 원칙

**결정:** `retries: 0`, `trace: on`, `screenshot: on` 항상 적용.

| 설정 | 값 | 이유 |
|------|----|------|
| retries | 0 | 실패를 숨기지 않는다. Lucky retry는 근본 원인 분석을 방해한다 |
| trace | on | 항상 캡처. 통과한 테스트에서도 이상 징후 포착 가능 |
| screenshot | on | 매 스텝 캡처. 실패 시점의 UI 상태 보존 |
| fullyParallel | false | 순차 실행. RCA 결과 읽기 편함 |

---

### Fault Isolation — 프론트/백엔드 소유권 결정

**배경:** 프론트엔드와 백엔드가 레포가 분리되어 있을 때,
네트워크/어서션 실패의 원인이 프론트엔드 버그인지 백엔드 버그인지 계약 불일치인지 먼저 판단해야 한다.

**결정:** Phase 3.5에서 API 트래픽을 캡처하고 소유권을 분류한 후 RCA를 시작한다.

**분류 기준:**

| 시그널 | 소유권 |
|-------|-------|
| 4xx + 응답이 API 스펙과 일치 | FRONTEND — 잘못된 요청 전송 |
| 4xx + 응답이 스펙과 불일치 | BACKEND — 잘못된 에러 형식 |
| 5xx | BACKEND — 서버 크래시 |
| 200 + 응답 shape 틀림 | BACKEND — 누락 필드 |
| 200 + 응답 shape 맞음 + UI 틀림 | FRONTEND — 렌더링 버그 |
| 200 + 타입과 불일치 | CONTRACT MISMATCH |

백엔드 소유권으로 분류된 버그는 별도 **Backend Bug Report** 형식으로 출력.
프론트엔드 코드를 패치해서 우회하지 않는다.

---

### RCA 원칙 — 증상이 아닌 근본 원인

**결정:** "element not found"는 루트 코즈가 아니다. 계속 "왜"를 물어야 한다.

```
BAD:  "element not found"
GOOD: "data-testid가 컴포넌트 리팩터링 때 제거됨"

BAD:  "timeout"
GOOD: "환경 변수 누락으로 API 호출이 pending 상태로 멈춤"

BAD:  "401 unauthorized"
GOOD: "두 요청이 동시에 발화될 때 토큰 갱신 로직에 race condition 존재"
```
